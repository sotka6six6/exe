# Есет — локальный запуск

## Быстрый старт (Windows)

1. Поставь Python 3.11 с [python.org](https://www.python.org/downloads/) —
   при установке обязательно отметь **"Add python.exe to PATH"**.
2. Скопируй `.env.example` в `.env` и впиши туда:
   - `TG_TOKEN` — токен бота от [@BotFather](https://t.me/BotFather)
   - `GROQ_API_KEY` — ключ с [console.groq.com/keys](https://console.groq.com/keys)
   - `ADMIN_ID` — твой Telegram user ID (узнать у [@userinfobot](https://t.me/userinfobot)).
     Без этого админ-панель (`/admin`, `/forget`, кнопки) будет доступна
     старому владельцу бота, зашитому в код, а не тебе.
3. Запусти `run.bat` двойным кликом.
   Первый раз он сам создаст venv и поставит зависимости — это может
   занять пару минут. Дальше запускается сразу.

Бот держит `run.bat` открытым — если упадёт с ошибкой, окно не закроется,
а покажет лог и предложит перезапустить.

## Деплой на Railway

Бот уже поддерживает Railway "из коробки" (см. `Procfile`, `DB_PATH` в
`rex_voice_v4_bot.py` использует `RAILWAY_VOLUME_MOUNT_PATH`, если он есть).

1. **Создай проект**: [railway.app](https://railway.app) → New Project →
   Deploy from GitHub repo (залей эту папку к себе в приватный репозиторий)
   или через Railway CLI: `railway init` → `railway up` из этой папки.
2. **Тип сервиса — Worker, не Web**: у бота нет HTTP-сервера (`run_polling`),
   поэтому Railway не должен ждать открытый порт. Если Railway всё же
   просит указать порт/health-check — в Settings сервиса отключи
   "Generate Domain" / health check, сервис всё равно будет работать как
   background worker благодаря `Procfile` (`worker: python rex_voice_v4_bot.py`).
3. **Переменные окружения** — Project → Variables, добавь (без кавычек):
   - `TG_TOKEN` — токен от @BotFather
   - `GROQ_API_KEY` — ключ с console.groq.com/keys
   - `ADMIN_ID` — твой Telegram ID
   - `GROQ_PROXY` — обычно НЕ нужен на Railway (сервера не в РФ, поэтому
     геоблок от Groq маловероятен даже если раньше был локально)
   `.env` файл на Railway не используется и не нужен — только Variables
   в дашборде.
4. **Постоянное хранилище БД**: без Volume файл `eset_memory.db`
   пересоздастся с нуля при каждом редеплое (Railway перезаписывает
   файловую систему контейнера). Чтобы память бота (юзеры, конфликты,
   рейтинги) сохранялась между деплоями:
   - Project → сервис → Settings → Volumes → Add Volume, точка монтирования
     любая, например `/data`.
   - Добавь переменную `RAILWAY_VOLUME_MOUNT_PATH=/data` (или Railway
     подставит её сам, если Volume подключён — проверь в Variables).
   - Код уже читает эту переменную сам, ничего в коде менять не надо.
5. **Деплой**: Railway сам подхватит `requirements.txt` и `Procfile`
   через Nixpacks и запустит `python rex_voice_v4_bot.py`. Логи — во
   вкладке Deployments → View Logs.



Группа-чат бот с характером ("Есет") — анализирует сообщения через LLM (Groq),
запоминает пользователей, их конфликты и отношения в SQLite, отвечает голосом
(edge-tts) или текстом в зависимости от настроения чата.

## Файлы

| Файл | Назначение |
|---|---|
| `rex_voice_v4_bot.py` | точка входа, обработчики Telegram, БД |
| `brain.py` | решение — отвечать или нет, в каком режиме |
| `analyzer.py` | глубокий LLM-анализ смысла сообщения |
| `dialogue.py` | диалоговый контекст (кто кому отвечает) |
| `vocab.py` | словари сленга/мата/фраз под настроение |
| `learner.py` | самообучение — что заходит в чате, что нет |
| `eset_memory.db` | база данных (создаётся сама при первом запуске) |

## Что было исправлено при доработке

- **Краш-баг**: реакция на "самокоманду" и подначки родителей
  (`topic in ("самокоманда", "мамапапа")`) обращалась к переменной
  `decision` до того как та вообще создавалась — валило
  обработку сообщения в этих темах. Перенесено после реального
  решения мозга, использует `dataclasses.replace`.
- **Утечка внутренностей в чат**: при любой необработанной ошибке бот
  писал в чат `"Сломался: " + str(e)` — то есть текст питоновского
  исключения (пути, детали API-ошибок Groq и т.п.) видели все участники
  группы. Теперь в чат уходит нейтральная фраза, а полная трассировка —
  только в лог на диске.
- **Зависание бота на сетевых вызовах**: анализ сообщения и генерация
  ответа дергают Groq API синхронно — раньше это блокировало *весь*
  event loop бота на время запроса (и retry на 429 — до ~35 секунд),
  то есть бот замирал сразу во всех чатах. Обёрнуто в
  `asyncio.to_thread`, event loop свободен.
- **`database is locked`**: SQLite открывался без таймаута из полутора
  десятков разных мест плюс фоновые asyncio-задачи — при параллельном
  доступе это падает с ошибкой блокировки. Добавлен общий `_connect()`
  с `WAL`-режимом и `busy_timeout`, все 52 места подключения переведены
  на него.
- **Зашитый Telegram ID админа**: раньше единственным человеком с
  доступом к `/admin` был ID оригинального разработчика. Вынесено в
  `ADMIN_ID` в `.env` — задай свой, иначе панель тебе не откроется.
- **Мёртвый модуль самообучения**: `learner.py` (оценка удачных фраз,
  генерация новых провокаций, подбор стиля под юзера) существовал в
  архиве, но ни разу не вызывался и даже не имел нужных таблиц в БД —
  вызов сразу упал бы с `no such table`. Таблицы добавлены в `init_db()`,
  фоновый цикл `learner_loop` запускается при старте, реакция чата на
  каждый ответ бота теперь реально фиксируется и используется в
  авто-провокациях (экономит вызовы LLM на уже проверенных фразах).
- **Краш на Python 3.14**: `python-telegram-bot==21.6` вызывает
  `asyncio.get_event_loop()` без работающего event loop — с Python 3.14
  это теперь `RuntimeError` вместо тихого создания нового loop, бот падал
  сразу на `run_polling()`. Обновлено до `python-telegram-bot==22.8`,
  где это официально исправлено под 3.14. Если у тебя уже стоит venv
  со старой версией — просто запусти `run.bat` ещё раз, `pip install`
  сам подтянет `22.8` по пину из `requirements.txt`.
- **`NameError: name 'COUNTER_DEEP' is not defined`**: в `vocab.py` эти
  фразы (`COUNTER_DEEP`, `SELFCMD_TEMPLATES`) реально есть, но их забыли
  добавить в строку импорта в `rex_voice_v4_bot.py` — оставались только
  в `except`-заглушке, которая никогда не срабатывала (импорт-то не
  падал). В итоге любое сообщение с темой "самокоманда" валилось с
  `NameError`, бот просто отвечал текстом ошибки в чат. Добавил оба
  имени в импорт, плюс сам `except` теперь пишет в лог, если что-то
  из vocab всё-таки не импортировалось — раньше он тихо проглатывал
  любую ошибку, из-за чего эта проблема была не видна до реального
  сообщения в чате.
- **Вторая утечка ошибок в чат, отдельно от первой**: у `ask_rex()` был
  свой собственный `except Exception as e: return "Сломалось нахуй: " + str(e)`
  — эта строка с текстом реальной ошибки (включая, например, `403
  Forbidden` от Groq) возвращалась как будто это нормальный ответ
  бота, отправлялась в чат и даже сохранялась в историю сообщений
  (из-за чего потом "гуляла" по контексту треда). Внешний try/except
  в `process_message`, который я чинил раньше, эту ошибку вообще не
  видел — `ask_rex` перехватывал её первым. Теперь при любой ошибке
  `ask_rex` возвращает `None`, как и остальные LLM-вызовы, и бот просто
  молчит на это сообщение вместо того чтобы репостить исключение.
- **`setMessageReaction` → 400 Bad Request**: Telegram принимает
  реакцию только если эмодзи входит в фиксированный список
  `telegram.constants.ReactionEmoji` (~70 штук) — это не любой
  юникод-эмодзи. В пулах реакций бота было несколько эмодзи вне этого
  списка (💀, 😤, 😏, 🙄, 😒, 👁 — обычный глаз вместо пары глаз 👀, 💤,
  😑), так что реакция на сообщение падала с 400 каждый раз когда
  выпадал один из них. Заменил на валидные аналоги и добавил проверку
  при старте, которая тихо вырежет из пулов любой невалидный эмодзи в
  будущем и напишет об этом в лог, вместо повторения той же ошибки.
- **TTS 403 (edge-tts)**: `Sec-MS-GEC` токен, который edge-tts шлёт на
  сервер Microsoft, генерировался по устаревшему алгоритму — сервер
  стал отвечать 403 на версии библиотеки старее ~7.x. Обновил
  `edge-tts` до `7.2.8`. Бот и раньше не падал на этой ошибке (падал
  обратно на текст), но голосовые не работали — теперь должны.

## Безопасность

- `.env` в `.gitignore` — токен и ключ никогда не попадут в git.
- Все SQL-запросы параметризованы (`?`) — SQL-инъекций в исходном коде
  не найдено, менять не пришлось.
- Админ-команды (`/admin`, `/forget`, callback-кнопки) везде проверяют
  `is_admin()` — обхода прав не найдено.

## Если видишь `HTTP 403 Forbidden` от api.groq.com

Это не баг в коде — Groq отклоняет сам запрос с твоим ключом. Обычно
значит одно из:
- ключ невалидный или отозван — проверь `GROQ_API_KEY` в `.env`, лишних
  пробелов/кавычек быть не должно;
- ключ создан в аккаунте с региональным или тарифным ограничением на
  этот конкретный конпоинт/модель;
- ключ создан для другого проекта в консоли Groq.

Проверить: [console.groq.com/keys](https://console.groq.com/keys) —
перевыпусти ключ и вставь заново. Бот не падает на этой ошибке (просто
молчит на конкретное сообщение), но пока ключ не рабочий, LLM-анализ и
ответы генерироваться не будут.
